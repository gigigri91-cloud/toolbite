FAQ

        

          

            Does this JWT Decoder verify signatures?▾
            
No. It only decodes header and payload for inspection. Signature verification requires your secret or public key and must happen in your backend/auth flow.

          

          

            Is my token uploaded to ToolBite servers?▾
            
No. Decoding runs locally in your browser and nothing is sent to ToolBite for processing.

          

          

            Why do I see an invalid token error?▾
            
Common causes are malformed Base64url segments, missing dots between segments, or non-JSON data in header/payload.

          

        

      

      

        
Advanced Usage & Examples


        

          

            
Scenario 1: Inspect `exp` and `nbf` timing quickly

            
Input token payload:
{
  "sub": "usr_1024",
  "iat": 1714900000,
  "nbf": 1714900300,
  "exp": 1714903900
}

Output check:
- nbf is 5 minutes after iat
- exp is 1 hour after iat
- token should be rejected before nbf

          


          

            
Scenario 2: Validate audience/issuer assumptions before backend checks

            
Decoded payload snippet:
{
  "iss": "https://auth.example.com",
  "aud": "toolbite-api",
  "scope": "read:profile"
}

Quick review:
- Expected issuer? yes
- Expected audience? yes
- Scope present? yes (still enforce on server)

          


          

            
Scenario 3: Debug malformed token structure

            
Input token:
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjMifQ

Decoder output:
- Missing third segment (signature)
- Treat token as structurally invalid
- Do not continue auth flow with partial token

          

        


        

          
Did you know? 🧠

          
A JWT can decode perfectly and still be unusable for auth if signature verification fails. Decode helps debugging; verify establishes trust.

        


        

          
Limitations

          
This tool does not verify signatures or key trust chains. Use it for inspection only, then run full verification in your backend auth layer.

        


        
How this differs from JSON Formatter: JWT Decoder parses token segments and claim structure; JSON Formatter validates generic JSON syntax without token semantics.

Why we do not verify signatures

        
Verifying a JWT requires the correct secret or public key. This tool only Base64url-decodes the first two segments so you can inspect claims (e.g. exp, sub) during development. Never treat decoded claims as authenticated without proper verification on your server.

Useful checks during inspection

        
Once decoded, compare the algorithm in the header, confirm issuer and audience in the payload, and check whether time-based claims such as exp, iat, and nbf match the environment you are testing. Those quick checks often explain auth bugs before you even step into backend logs.

        
Safer troubleshooting habit

        
Use local decode tools for inspection, but avoid pasting live production tokens into tickets, chat threads, or cloud tools. If you need to share an example with teammates, redact or regenerate the token first and keep real verification inside your application code or auth middleware.
